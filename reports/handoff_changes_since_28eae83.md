## 已完成

### 1. What-if 模擬按鈕
- Dashboard 與風險訂單管理的「模擬」統一導向 What-if 面板，並帶入該列的運送方式、地區、計畫天數、產品價格、數量、客群、市場與日期。
- 修正風險訂單清單按鈕無作用：原先把 JSON 直接放入 HTML `onclick`，雙引號會截斷屬性；改為安全的 `data-*` 屬性與事件監聽。
- 模擬器會補入下拉選單中尚未存在的地區或運送方式，並在頁面尚未準備好時重試，避免切頁後無法帶入。

### 2. 收益模型洩漏守門
- 單筆收益預測改採 `serving_artifacts.json` 與 `profit_feature_manifest.json` 的部署特徵契約，不再使用可能殘留舊欄位的 `profit_feature_schema.json`。
- 洩漏守門改檢查實際部署契約；舊 schema 若仍含禁止欄位，會標示為 stale，但不會誤判已部署模型為 FAIL。
- 模型診斷頁會顯示守門採用的契約來源與舊 schema 狀態。

### 3. 風險分級統一
- 新增 `core/risk_policy.py` 作為唯一規則來源：Low < 30%、Medium 30% 至未滿 70%、High >= 70%。
- 訓練輸出、上傳預測、單筆 What-if、既有 predictions/decision CSV 的 API 回傳、最佳化器、主管解釋器與 Dashboard 呈現均改用同一套規則。
- 既有 CSV 不需重訓或重建；API 載入時會依 `p_late` 即時重新標示。
- 最佳化的候選門檻維持 30%。它是經濟篩選入口，和 High 標籤 70% 不同：Medium 訂單若有正期望效益仍可參與最佳化。

### 4. 最佳化候選池限制
- 流程仍為：`p_late >= risk_threshold` -> `預期延遲損失 - 升級成本 > 0` -> 若超過 500 筆，保留個別淨效益最高的 500 筆 -> 在這些候選內依預算做 MILP 組合最佳化。
- 真實單次匯入不超過 500 筆時不會截斷，最佳化涵蓋完整候選集；以全歷史資料直接執行時，500 筆上限屬於運算近似，不應宣稱全體訂單的全域最佳解。
