# EDIS UI 工作流程檢視：資深供應鏈經理視角

檢視角度：具備 20 年以上物流、承運商管理、SLA 控管、異常升級處理與預算核准經驗的資深供應鏈經理。

目標：讓 EDIS 更像一套「決策系統」，而不只是分析儀表板。以下 10 個主題都列出目前問題、需要移除的部分、需要新增的功能、需要調整的文字或互動，以及應該放在哪一個頁面。

---

## 1. 讓主要工作流程更清楚

### 目前問題

目前 Dashboard 同時呈現太多工具：門檻值設定、CSV 上傳、KPI、主管摘要、預算情境表、風險訂單表，以及最佳化控制。對經理人來說，不一定能立刻知道應該先看哪裡、接著做什麼，以及整個操作順序是什麼。

### 需要移除

- 移除 Dashboard 右下角的小型最佳化輸入區，避免跟「最佳化調度」頁面重複。
- 如果某個決策操作已經存在於「最佳化調度」頁面，就不要在 Dashboard 重複放一套類似控制項。
- 在使用者理解商業決策前，不要先呈現太多技術設定，例如模型門檻、F1 等。

### 需要新增

- 在 Dashboard 上方新增工作流程進度列：
  1. 確認資料
  2. 檢視風險
  3. 選擇策略
  4. 執行最佳化
  5. 核准行動方案
  6. 追蹤實際結果
- 在 Dashboard 新增一個明確的主要 CTA：
  - `檢視高風險訂單`
  - 次要 CTA：`執行最佳化調度`
- 為每個流程步驟新增狀態提示：
  - 資料已載入
  - 模型啟用中
  - Manager 是否已登入
  - 最佳化尚未執行 / 已執行
  - 行動方案尚未核准 / 已核准

### 頁面調整

- 將 `Dashboard` 定位為高階主管的起始頁與總覽頁。
- 將詳細最佳化控制完整移到 `最佳化調度` 頁面。
- 保留 `風險訂單列表` 作為訂單明細檢視與篩選頁。

## 2. 用商業語言重新包裝門檻值設定

### 目前問題

Threshold slider、F1 mode、cost mode 對資料團隊有用，但對供應鏈經理偏技術。經理人更在意的是商業取捨：要抓更多可能延遲的訂單，還是控制成本並接受部分漏判風險。

### 需要移除

- 不要在 Dashboard 快速操作中把 `F1` 當成主要可見標籤。
- 不要把門檻調校面板放在 Dashboard 第一眼可見的位置。
- 不要讓數字門檻值成為經理人看到的第一個決策控制。

### 需要新增

- 新增商業策略選項：
  - `積極保護 SLA`：抓更多可能延遲訂單，但會花更多預算
  - `平衡營運`：在抓到風險與避免誤報之間取得平衡
  - `成本控管`：減少預算支出，但接受較高漏判風險
- 為每個策略新增簡單影響預覽：
  - 預估可捕捉的延遲訂單數
  - 預估誤報數
  - 預估升級成本
  - 預估淨效益

### 需要調整

- 將 `平衡模式 (F1)` 改成 `平衡營運`。
- 將 `成本最低模式` 改成 `成本控管`。
- 數字門檻值仍可保留，但放進 `進階模型設定` 展開區。

### 頁面調整

- Dashboard：只顯示商業策略選擇器。
- 模型效能頁：保留技術性的 threshold / F1 細節。
- 最佳化調度頁：只在進階設定中允許覆寫 threshold。

## 3. 解決預算建議數字互相衝突的問題

### 目前問題

主管摘要可能建議非常大的預算，而情境分析表又建議較小的預算。這會讓經理人在向財務或高層提案時感到困惑：到底該拿哪個數字當正式預算需求？

### 需要移除

- 移除或隱藏多個互不相關的 `recommended_budget`，除非畫面明確解釋每個數字的意義。
- 移除模糊標籤，例如 `Budget Ask`，如果它和情境分析建議的預算不是同一個概念。

### 需要新增

- 新增一張統一的 `建議預算` 卡片。
- 新增計算基礎說明：
  - `完整正 ROI 機會`：所有正 ROI 訂單都升級時需要的總預算
  - `實務建議情境`：目前測試情境中最適合採用的預算
  - `目前輸入預算`：使用者當下輸入或選擇的預算
- 新增預算比較表：
  - 預算金額
  - 選中訂單數
  - 預估避免罰金
  - 升級成本
  - 淨效益
  - 與前一個情境相比的邊際效益

### 需要調整

- Dashboard 預算文字：
  - 原文：`ROI 為正訂單的建議升級預算`
  - 建議改為：`若升級所有正 ROI 訂單所需的完整機會預算`
- 情境建議文字：
  - 原文：`建議至少保留 USD $10,000`
  - 建議改為：`目前測試情境中最佳方案：USD $10,000`

### 頁面調整

- Dashboard：顯示一張含定義的主管預算建議卡。
- 最佳化調度頁：顯示更詳細的情境比較與預算敏感度分析。

## 4. 加入清楚的核准與執行流程

### 目前問題

系統會建議哪些訂單應該升級，但最佳化後沒有明確下一步。經理人需要核准、匯出、分派給物流團隊，或追蹤行動是否已執行。

### 需要移除

- 不要讓使用者誤以為「最佳化結果」就是流程終點。
- 不要讓流程停在 `建議升級訂單`。

### 需要新增

- 在最佳化結果後新增操作按鈕：
  - `核准方案`
  - `匯出 CSV`
  - `指派給物流團隊`
  - `標記為已執行`
- 新增方案狀態：
  - 草稿
  - 已核准
  - 已匯出
  - 已執行
  - 已回填實際結果
- 新增核准摘要：
  - 核准主管
  - 核准時間
  - 核准預算
  - 核准訂單數
  - 預估節省金額

### 需要調整

- 將 `調度預估成效分析` 改成 `最佳化行動方案`。
- 將 `升級訂單明細與預估省下金額` 改成 `建議升級行動清單`。

### 頁面調整

- 最佳化調度頁：新增核准、匯出、執行相關功能。
- Dashboard：顯示最新已核准方案狀態。
- 建議新增頁面：`行動方案歷史紀錄`。

## 5. 為風險訂單加入營運所需欄位

### 目前問題

目前風險訂單列表主要顯示訂單 ID、運送模式、地區、延遲機率、風險等級與預測是否正確。這些資訊不足以讓供應鏈經理採取實際行動。

### 需要移除

- 當使用者查看的是未來訂單或尚未知道實際結果的訂單時，主風險列表不應顯示 `預測正確？`。
- 將預測正確與否移到模型效能頁或歷史驗證檢視。

### 需要新增

在 `風險訂單列表` 與最佳化行動清單中加入以下欄位：

- 承諾到貨日
- 距離承諾到貨日剩餘天數
- 出貨倉庫 / 配送中心
- 目的地國家 / 城市
- 承運商或配送夥伴
- 客戶優先級 / SLA 等級
- 商品類別
- 訂單價值或罰金暴露金額
- 目前出貨狀態
- 建議行動負責人

### 需要調整

- 調整表格分組方式，支援營運掃描：
  - 依地區
  - 依承運商
  - 依 SLA 等級
  - 依倉庫
- 新增篩選條件：
  - 承諾到貨日期區間
  - SLA 等級
  - 承運商
  - 倉庫
  - 行動狀態

### 頁面調整

- 風險訂單列表頁：作為主要營運檢視表。
- 最佳化調度頁：被選中的行動清單應包含相同營運欄位。
- 模型效能頁：保留實際值 vs 預測值相關欄位。

## 6. 讓 AI 助理更有引導性且更可信

### 目前問題

AI 助理目前暗示使用者可以問任何問題，這對商業系統太寬泛。而且目前整合測試顯示，off-topic guardrail 尚未穩定觸發。

### 需要移除

- 移除 placeholder `提出任何問題`。
- 移除 AI 可以回答無關問題的暗示。

### 需要新增

- 新增引導式 prompt chips：
  - `摘要今天最高物流風險`
  - `解釋此地區風險為何偏高`
  - `建議這批訂單需要多少預算`
  - `產生主管簡報摘要`
  - `列出最優先升級訂單`
- 新增 AI 信任邊界說明：
  - `AI 僅使用去識別化物流資料，且只回答物流風險、調度最佳化與模型解釋相關問題。`
- 在 AI 回答中加入資料來源與假設：
  - 使用的 KPI
  - 使用的 threshold
  - 使用的 budget
  - 資料時間戳

### 需要調整

- 將 AI 助理從自由聊天改成決策簡報模式。
- 如果使用者問 off-topic 問題，顯示：
  - `此助理僅支援物流風險、最佳化調度與模型解釋相關問題。`

### 頁面調整

- AI 助理頁：新增 prompt chips 與信任邊界說明。
- LLM 設定頁：保留 provider、model、API key 設定。
- Backend/API：修正 off-topic guardrail，讓整合測試通過。

## 7. 避免業務使用者誤操作模型重訓

### 目前問題

最佳化調度頁包含月份誤差診斷與模型重訓控制。這些功能很強，但對非技術經理人有風險，因為採用新模型可能影響整個系統判斷。

### 需要移除

- 移除一般 Manager 流程中的直接 `採用新模型`，除非有明確治理步驟。
- 移除沒有充分說明的 `選擇要排除的問題特徵` 操作，因為經理人可能不知道統計後果。

### 需要新增

- 新增模型治理狀態：
  - 目前正式模型
  - 候選模型
  - 等待驗證
  - 已由資料科學 / Admin 核准
- 採用模型前新增確認說明：
  - 哪些內容改變
  - 哪些特徵被移除
  - 指標影響
  - 商業風險
  - 是否可回滾
- 新增角色分工：
  - Manager 可以提出重訓請求
  - Data Scientist / Admin 才能核准正式採用模型

### 需要調整

- 將 `開始重訓模型` 改成 `提出模型重訓審查`。
- 將 `採用新模型` 改成 `提交候選模型等待核准`，除非系統新增 Admin 角色。

### 頁面調整

- 將重訓流程從最佳化調度頁移到模型效能頁，或新增 `模型治理` 頁面。
- 最佳化調度頁可以保留月份診斷作為風險訊號，但不要在該頁直接改正式模型。

## 8. 將技術型 RBAC 頁改成業務權限頁

### 目前問題

目前 RBAC 頁顯示 API endpoint 權限表。這對開發者有用，但對經理人來說，他們更想知道每個角色在應用程式中可以做什麼。

### 需要移除

- 從主要使用者介面移除 API endpoint 權限矩陣。
- 將 endpoint 細節移到開發者文件或進階區塊。

### 需要新增

- 新增業務權限矩陣：
  - 查看風險 Dashboard
  - 篩選風險訂單
  - 上傳待預測 CSV
  - 回填實際結果
  - 執行最佳化
  - 核准行動方案
  - 設定 AI
  - 提出模型重訓請求
  - 採用正式模型
- 新增角色說明：
  - Viewer
  - Logistics Manager
  - Data Scientist / Admin，如果後續新增

### 需要調整

- 將頁面標題從 `RBAC 角色存取權限矩陣` 改成 `使用者角色與權限`。
- 將內容語言從 API 導向改成工作流程導向。

### 頁面調整

- RBAC 頁：改成業務權限矩陣。
- README 或開發者附錄：保留 API endpoint 權限矩陣。

## 9. 加入資料新鮮度與資料來源信任訊號

### 目前問題

Top bar 目前顯示 `Mock Data`。對供應鏈經理來說，這會讓人不確定資料到底是 demo、上傳資料、正式資料、過期資料，還是缺少實際到貨結果。

### 需要移除

- 移除只顯示 `Mock Data` 的單一狀態。
- 不要隱藏目前使用的是預設驗證資料還是使用者上傳資料。

### 需要新增

- 新增資料狀態卡：
  - 資料模式：demo / uploaded / production
  - 最新上傳時間
  - 訂單數
  - 資料日期區間
  - 是否有實際到貨結果
  - 目前套用的 prediction threshold
  - 模型版本
- 新增警示狀態：
  - `目前使用 Demo Data`
  - `尚未提供實際結果`
  - `資料已超過 X 天未更新`
  - `模型未啟用`

### 需要調整

- 將 top bar 狀態從 `Mock Data` 改成更具體的狀態：
  - `Demo Data · 27,078 orders · Model v1.0`
  - 或 `Uploaded Data · 4,238 orders · No actual outcomes`

### 頁面調整

- Top bar：顯示簡短資料模式標籤。
- Dashboard：在上方加入完整資料狀態卡。
- AI 助理：每次產生主管摘要時，包含資料模式與資料時間。

## 10. 加入決策後的商業成效追蹤

### 目前問題

系統會估算節省金額，但沒有呈現過去建議是否被接受、是否已執行，以及實際結果是否符合預期。沒有這個回饋循環，經理人很難長期信任系統。

### 需要移除

- 不要只把 `expected savings` 當作唯一成功指標。
- 不要在同一張表混合歷史實際結果與未來預測，除非標示非常清楚。

### 需要新增

- 新增行動歷史：
  - 方案 ID
  - 核准預算
  - 選中訂單
  - 執行狀態
  - 實際避免延遲訂單數
  - 實際避免罰金
  - 實際結果與預估節省差異
- 新增決策後 KPI：
  - 預估 SLA 改善
  - 實際 SLA 改善
  - 誤報數
  - 漏判延遲數
  - 執行後 ROI
- 新增回饋流程：
  - 上傳實際結果
  - 比較預估 vs 實際
  - 更新模型效能

### 需要調整

- 將 `模型效能` 頁加入商業成效區塊，不只顯示統計指標。
- 將 Dashboard 從單次快照改成「目前狀態 + 最近趨勢」：
  - 目前風險
  - 最新核准方案
  - 上一次方案的實際結果

### 頁面調整

- 建議新增頁面：`成效追蹤` 或 `行動歷史`。
- Dashboard：顯示最新方案與實際結果摘要。
- 模型效能頁：將統計模型效能與商業成效分開呈現。

## 建議調整後的頁面結構

### Dashboard

- 資料新鮮度 / 資料狀態
- 高階主管決策卡
- 工作流程進度列
- 核心 KPI
- 建議預算摘要
- 最新行動方案狀態
- 前往風險列表或最佳化調度的 CTA

### 風險訂單列表

- 營運用篩選條件
- 含行動欄位的風險訂單表
- 單筆訂單解釋 modal
- 可將訂單加入草稿行動方案

### 最佳化調度

- 商業策略選擇器
- 預算情境比較
- 執行最佳化
- 建議行動清單
- 核准 / 匯出 / 執行流程

### AI 助理

- 引導式主管簡報 prompt
- 僅限物流領域的 guardrail
- 回答中顯示資料來源與假設

### 模型效能 / 模型治理

- 技術指標
- 錯誤分析
- 月份 drift / error diagnosis
- 重訓請求與核准流程

### 區域風險地圖

- 區域熱力圖
- 點選地區後連到篩選後的風險訂單
- 地區層級預算建議

### 使用者角色與權限

- 業務權限矩陣
- 開發者 API 權限表移到進階或文件

### 成效追蹤

- 已核准方案
- 執行狀態
- 實際 vs 預估節省
- SLA 影響
- 實際結果回填

## 建議實作優先順序

1. 先修 Dashboard 的工作流程清楚度與資料狀態。
2. 解決預算建議文字與數字定義衝突。
3. 新增最佳化方案核准 / 匯出 / 執行流程。
4. 為風險訂單加入營運欄位與篩選條件。
5. 將 AI 助理改成引導式，並修正 off-topic guardrail。
6. 將模型重訓移到有治理流程的頁面。
7. 將 RBAC 頁改成業務權限矩陣。
8. 加入行動方案執行後的成效追蹤。
